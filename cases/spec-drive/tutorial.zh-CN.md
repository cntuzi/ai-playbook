# 规格驱动开发 — 新手教程

> 从零理解"如何让 AI 参与软件开发的全链路，而不仅仅是写代码"。
> 适用于：新团队成员入门 / 其他项目参考采用。

---

## 第一章：解决什么问题

### AI Coding 的现状

AI 工具能写代码了，但软件开发不只是写代码。

```
产品给了 PRD → 开发者让 AI 写代码 → 能跑吗？ → 样式对吗？ → 接口对吗？
                                         ↓
                                    不知道，祈祷吧
```

| 问题 | 后果 |
|------|------|
| AI 不知道项目用什么技术栈 | 生成的代码和现有架构不兼容 |
| AI 不知道设计稿长什么样 | 写出来的 UI 和 Figma 差十万八千里 |
| AI 不知道后端接口定义 | 请求参数瞎猜，字段名对不上 |
| 没人追踪"做了什么、还剩什么" | 进度全靠口头问 |
| 需求变了，没人知道影响多大 | 改了 A，B 和 C 也要改，但忘了 |

### 核心思路：在 PRD 和代码之间加一层

**结构化的"功能规格"，是人和 AI 的共同语言。**

```
PRD (给人看的自然语言)
    ↓ 解析提取
功能规格 (结构化，人 + AI 都能读)        ← 这是核心
    ↓ AI 读取后执行
代码 (给编译器看)
```

一份功能规格把散落各处的信息**合并成一个文件**：

| 传统方式 | 功能规格方式 |
|---------|------------|
| 需求看 PRD 第 N 页 | 结构化需求列表 |
| 设计看 Figma 链接 + 口头确认 | 设计约束 + Figma 节点引用 |
| 验收标准散落各处 | 明确的通过/不通过条件 |
| API 看 Swagger + 飞书消息 | 接口引用 |
| 埋点看 Excel | 结构化埋点定义 |
| 国际化看翻译表格 | 按功能分组的文案 |
| 状态场景靠脑补 | 状态场景穷举表 |

AI 读取这个文件就获得完整上下文，不需要人反复解释。

## 第二章：系统全景

### 三个阶段

```
阶段 1：生成规格
    从 PRD + Figma + API 文档 → 自动生成功能规格 + 任务计划
    (30 分钟完成传统 10+ 小时的工作)

阶段 2：调度执行
    分析任务依赖 → 找出哪些可并行 → 分派 AI 开发 → 监控进度

阶段 3：自治开发
    AI 在隔离环境中：读规格 → 写代码 → 编译验证 → 自动合并 → 更新进度
```

### 一个版本是怎么跑的

```
1. 产品交付 PRD + Figma + Swagger

2. 自动生成规格骨架
   → 15 个功能规格文件、双端任务计划、后端依赖、国际化文案、埋点定义

3. 创建版本分支

4. 分析依赖图，规划并行批次
   → 第一批: 6 个无依赖任务同时开工
   → 第二批: 3 个任务，依赖第一批的结果
   → 阻塞: 等后端 API 就绪后自动解除

5. 开发中需求变了？
   → 通过依赖索引秒级定位影响范围 → 自动生成返工清单

6. 实时看板
   → 谁完成了、谁阻塞了、阻塞原因是什么
```

### 核心文件一览

```
{version}/                           一个版本的全部规格
├── config.yaml                      版本配置（功能清单、资源路径、依赖索引）
├── features/                        功能规格（每个功能一个文件）
│   ├── F01-search.yaml                做什么 + 怎么约束
│   ├── F02-login.yaml
│   └── ...
├── tasks/                           任务计划
│   ├── ios.md                         iOS 任务 + 状态
│   ├── android.md                     Android 任务 + 状态
│   ├── backend.md                     后端 API 依赖
│   └── shared.md                      跨端前置依赖
├── figma-index.md                   Figma 设计稿节点索引
├── i18n/strings.md                  国际化文案
├── CHANGELOG.md                     变更记录
├── DASHBOARD.md                     进度看板（从任务文件聚合）
├── implementation/                  实现方案
│   └── F01-search/
│       ├── design.md                  通用方案（跨平台共享）
│       └── ios.md                     平台细化方案
└── _logs/                           执行日志
```

## 第三章：核心概念

### 3.1 功能规格文件 — 一个功能的全部定义

两大类字段：**做什么** + **怎么约束**。

**做什么（What）** — 可自动从 PRD 提取：

```yaml
id: F06
name: 互动拦截方案二期
description: |
  拦截弹窗升级：静态提示→动态倒计时

requirements:
  - id: R01
    desc: 拦截弹窗文案改为动态倒计时 MM:SS

acceptance_criteria:
  - id: AC01
    type: ui
    desc: 拦截弹窗居中显示，深灰圆角卡片
```

**怎么约束（Constraint）** — 需人工或半自动补充：

```yaml
# 视觉约束 — 必须做什么、禁止做什么
ui_contract:
  required:
    - 自定义居中弹窗（非系统 Alert）
  forbidden:
    - UIAlertController
  key_tokens:
    dialog_bg: "#17171A"
    corner_radius: 20

# 状态场景穷举 — 防止遗漏边界情况
state_matrix:
  - id: S01
    name: 能量弹窗-倒计时中
    figma_node: "377:1342"            # 对应 Figma 设计稿节点
    trigger: 聊天轮次耗尽
    expected: 居中弹窗，倒计时递减
  - id: S02
    name: 次数超限
    trigger: 当日广告次数达上限
    expected: Toast"今日次数已达上限"

# 量化尺寸 — 拒绝"肉眼差不多"
pixel_baseline:
  dialog_corner_radius: 20
  button_gap: 24
```

**为什么需要约束？** 没有它，AI 只知道"做一个弹窗"，不知道用什么组件、什么颜色、什么尺寸。结果就是写完发现不对，反复返工。有了它，一次到位。

**实际数据**：v1.3 中唯一有完整约束的功能（F06），返工次数为 0。其余 14 个没有约束的功能，累计产生了 ~20 次返工。

### 3.2 任务文件 — 进度的唯一状态源

```
tasks/ios.md 中的一条任务：

| T05 | character | 故事卡片菜单及置顶 | F04 | P1 | 🟢 | - |
```

状态流转：

```
🔴 待开始 → 🟡 进行中 → 🟢 已完成
                              ↓ 需求变了
                          🔵 需返工 → 🟡 → 🟢
```

关键规则：**任务文件是唯一状态源**。AI 在这里读写状态，进度看板从这里自动聚合。不需要手动维护看板。

### 3.3 变更记录 — 需求变了怎么办

不是口头说一句，而是：

```
1. 记录变更（编号 CR-001、CR-002...）

2. 通过依赖索引自动分析影响范围：
   这个 API 变了 → 影响功能 F06 → 影响任务 T10 (iOS) + T10 (Android)

3. 生成返工清单（checklist），逐项完成并标记

4. 全部完成 → 变更记录标记 ✅
```

**价值**：需求变了不会漏改。v1.3 经历了 5 次需求变更，每次都通过变更记录完整追踪。

### 3.4 依赖索引 — 反向查询表

版本配置中维护三张反向查询表：

| 查询方向 | 用途 |
|---------|------|
| API 端点 → 功能列表 | API 变了，秒级定位哪些功能受影响 |
| Figma 节点 → 功能列表 | 设计改了，秒级定位哪些功能受影响 |
| 功能 → 后端任务列表 | 功能变了，找到对应的后端依赖 |

### 3.5 并行批次 — 不是按顺序做，是按依赖做

分析任务间的依赖关系，找出哪些可以同时进行：

```
第一批: T01, T06, T07, T11  (无依赖，同时开工)
第二批: T02, T08             (依赖第一批的结果)
第三批: T03                  (依赖第二批)
阻塞:   T09                  (等后端 API，就绪后自动解除)
```

### 3.6 AI 自治开发循环

AI 在隔离的 git 分支中独立完成一个任务的全流程：

```
读取功能规格 → 收集上下文(设计稿/API/国际化)
→ 锁定任务(🔴→🟡) → 设计方案 → 写代码 → 编译验证
→ 代码审查 → 合并到版本分支 → 更新状态(🟡→🟢)
→ 下一个任务 or 退出
```

不需要人手动启动每一步。AI 自己决定做什么、怎么做、做完了更新状态。**人的角色是审查和决策。**

### 3.7 执行日志 — 事后可追溯

每次 AI 工作都写日志。几种常见类型：

| 类型 | 什么时候产生 |
|------|------------|
| 功能开发 | 完成一个任务 |
| 走查修复 | 人工发起代码/UI 审查 |
| 截图驱动对齐 | 用户发截图，AI 对比 Figma 修复偏差 |
| 单点修复 | 发现并修复一个 bug |
| 文档同步 | PRD / API 文档有更新 |
| 需求变更 | 记录一次 CR |
| 工作流回顾 | 聚合分析多轮日志，发现改进点 |

日志的价值：回答"**为什么这个模块改了 3 轮才收敛**" — 是规格缺失？门禁跳过？还是设计本身有歧义？

## 第四章：看一个真实案例

> 用 MooX v1.3 的真实数据走一遍。不用动手，只看。

### 看版本配置

`moox/1.3/config.yaml` — 15 个功能、Figma 文件 key、10 个 Swagger 文件、依赖索引。**整个版本的"目录"。**

### 看一个功能规格

`moox/1.3/features/F06-intercept-phase2.yaml`（300 行）：

- 6 条需求，其中 R03 标注了 `[CR-003] 已删除`（需求变更可追溯）
- 6 条验收标准
- 视觉约束：精确到颜色 `#17171A`、圆角 `20px`、蒙层透明度 `0.8`
- 6 个状态场景，每个绑定 Figma 节点
- 5 条埋点、2 个 API、国际化引用

**AI 开发这个功能时读取的全部上下文，都在这一个文件里。**

### 看任务文件

`moox/1.3/tasks/ios.md` — 18 个任务，16 个已完成、1 个进行中、1 个等后端。每个已完成任务有完成时间、合并 commit、实现摘要。**这是 iOS 端的进度看板。**

### 看变更日志

`moox/1.3/CHANGELOG.md` — 5 次变更记录。CR-001 一次改了 7 个功能，CR-003 删了一个模块又新增一个功能。每个变更有影响范围和传播 checklist。

### 看执行日志

`moox/1.3/_logs/` — 27 个日志文件，记录了 v1.3 的全部 AI 工作历史。

## 第五章：如何在自己的项目中使用

### 最小起步（10 分钟）

不需要一步到位。从最有价值的部分开始。

#### 阶段 1: 只写功能规格文件

创建一个目录，为每个功能写一个 YAML：

```yaml
id: F01
name: 用户登录
description: 支持手机号 + 验证码登录

requirements:
  - id: R01
    desc: 输入手机号，点击发送验证码
  - id: R02
    desc: 输入验证码，点击登录

acceptance_criteria:
  - id: AC01
    type: interaction
    desc: 验证码 60 秒倒计时，期间按钮不可点

state_matrix:
  - id: S01
    name: 未输入
    trigger: 打开登录页
    expected: 手机号输入框获焦，登录按钮灰色不可点
  - id: S02
    name: 已输入手机号
    trigger: 输入 11 位手机号
    expected: 发送验证码按钮可点
  - id: S03
    name: 倒计时中
    trigger: 点击发送验证码
    expected: 按钮变为 "Ns 后重新发送"，不可点
```

仅此一步，AI 工具就能读取这个文件获得完整上下文。**不需要安装任何东西，不需要学任何命令。**

#### 阶段 2: 加上任务文件

创建一个 Markdown 表格追踪进度：

```markdown
| ID | 任务 | 功能 | 状态 | 依赖 |
|----|------|------|------|------|
| T01 | 登录页 UI | F01 | 🔴 | - |
| T02 | 验证码发送 | F01 | 🔴 | T01 |
| T03 | 首页列表 | F02 | 🔴 | - |
```

完成一个任务就把 🔴 改成 🟢。简单直接。

#### 阶段 3: 对复杂 UI 加约束（可选）

只在 UI 复杂的功能上投入，简单功能不需要：

```yaml
ui_contract:
  required:
    - 自定义输入框组件
  forbidden:
    - 系统默认 TextField 样式
  key_tokens:
    input_height: 48
    corner_radius: 8
    brand_color: "#FF6B00"
```

### 完整接入

如果想用完整的自动化能力（规格自动生成、任务调度、并行开发）：

```
1. 参考本仓库的目录结构搭建
2. 配置版本信息（Figma key、Swagger 路径）
3. 用生成工具从 PRD + Figma + API 自动生成规格骨架
4. 人工确认 + 补充复杂功能的视觉约束
5. 启动任务调度 → AI 自治开发
```

具体的自动化命令和编排协议见仓库内的 `.claude/commands/` 目录。

### 裁剪指南 — 什么可以不要

| 组件 | 何时可以省略 |
|------|------------|
| 视觉约束 / 量化尺寸 | 非 UI 功能、原型阶段 |
| Figma 集成 | 没有 Figma 的项目 |
| 双端并行 | 单端项目 |
| 变更记录追踪 | 需求稳定的小项目 |
| 执行日志 | 不需要事后分析时 |
| 进度看板自动聚合 | 任务少于 10 个时 |
| git 隔离分支 | 单人开发时 |

**核心不可省的只有两个：功能规格文件 + 任务文件。** 其余按需叠加。

## 第六章：常见问题

### Q: 和 Jira / Linear 有什么区别？

Jira 管"谁做什么"。功能规格管"做什么 + 怎么约束 + AI 怎么执行"。

Jira ticket 没有 Figma 节点绑定、没有状态场景穷举、没有视觉红线。AI 读 Jira 只能知道"做一个登录页"，不知道用什么组件、什么颜色、什么状态。

**两者互补，不冲突。**

### Q: 需要会写 YAML 吗？

不需要精通。YAML 就是带缩进的 key-value。看一个例子就能模仿。

而且大部分内容可以从 PRD 自动生成，人只需要确认和补充少量字段。

### Q: 绑定特定编程语言吗？

不绑定。功能规格描述的是"做什么"，不是"用什么语言做"。

Swift、Kotlin、React、Flutter、Go — 都能用。需要适配的只有技术栈约束字段和编译命令。

### Q: 只有我一个人开发，值得用吗？

只用功能规格 + 任务文件，10 分钟搭起来。它的价值不是"管理团队"，而是**给 AI 提供结构化上下文**。一个人开发时 AI 是你的搭档，搭档需要读懂你的需求。

### Q: 和 .cursorrules / copilot-instructions.md 什么关系？

那些是"AI 编码风格配置" — 告诉 AI 用什么语法写代码。

功能规格是"AI 工作上下文" — 告诉 AI 做什么功能、做到什么程度、有哪些约束。

```
.cursorrules       → "用 Swift 5，MVVM 架构，SnapKit 布局"
功能规格            → "做一个登录页，有 3 个状态，用这些颜色，禁止用系统 Alert"
任务调度            → "先做 T01，再做 T02，T03 等后端 API"
```

层次不同，配合使用。

### Q: 规格写得太详细会不会浪费时间？

看功能复杂度。我们用 `ui_weight` 分三档：

| 档位 | 适用范围 | 需要写多少约束 |
|------|---------|--------------|
| 重度 UI | 弹窗、面板、新页面 | 视觉约束 + 量化尺寸 + 状态场景 |
| 轻度 UI | 列表项、文案改动 | 状态场景就够 |
| 纯逻辑 | 埋点、API 对接 | 不需要约束 |

**在返工频率高的地方投入约束，在返工频率低的地方省略。** 不是一刀切。
